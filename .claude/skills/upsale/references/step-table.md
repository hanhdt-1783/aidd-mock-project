# Upsale Skill — Step I/O Table

Most steps are driven by an LLM actor reading its `references/` prompt and writing the artifact in the shape declared by its `templates/` file. **Steps 1, 5a, 5c, and 7** are exceptions — they run as deterministic Python scripts (`scripts/detect_sdd.py`, `scripts/combine_proposals.py`, `scripts/phase_d_prep.py`, `scripts/apply_verdicts.py`), invoked by the orchestrator via Bash. Most LLM steps run as `Task(researcher)` subagents; **Step S** is the exception — it runs at orchestrator level (composes the `/tkm:scan-codebase` skill via the `Skill` tool, then spawns parallel `Explore` agents directly), because researcher subagents lack the `Agent` tool needed for fan-out. **Steps 3.1.NN, 3.2.NN, 3.3.NN (12 biz), 4.1.NN, and 4.2.NN (14 tech)** are also orchestrator-level fan-outs (one researcher subagent per item, batched ≤10 globally across the active fan-out phase).

The `references/` file is the source of truth for that step's procedure; the matching `templates/` file is the structural contract for the writer. For steps 3.3.NN and 4.2.NN, each subagent reads two files: its per-aspect reference (`references/business/03-improvement/<NN>-<slug>.md` or `references/technical/02-improvement/<NN>-<slug>.md` — Goal + use-context overrides + intake gate) AND the shared contract (`references/business/03-improvement.md` or `references/technical/02-improvement.md` — Shared rules + Ownership map). Fan-out items (3.1.NN, 3.2.NN, 3.3.NN (×12 biz), 4.1.NN, 4.2.NN (×14 tech), 6.<NN>-<slug>) are dispatched in batches of ≤10 concurrent globally across the active fan-out phase (see `references/orchestrator-protocol.md` → `## Phase B-discovery`, `## Phase B-research`, `## Phase B-improvement`, `## Phase D — Validation`). Step 5c (the phase-d-prep dispatcher) is a SINGLE-agent step, NOT a fan-out — it sequentially reads `<track>/<step-folder>/<NN>-<aspect-id>.md` files (matching the proposal item's `<!-- aspect-id: <slug> -->` rollup-heading comment to the aspect slug) and writes one payload JSON per surviving item. Step 6 then receives only file paths inline; the validator opens its payload via `Read({payload_path})` as its first action.
